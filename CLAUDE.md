## Current State

The MVP is feature-complete. `app/` is the Electron app (forked from hello-pear-electron). Build-order Steps 1–4 done: discovery, **pointers-heartbeat gossip** (Protomux channel; a single `pointers` message carries liveness + country + Chord succ/pred), live sorted ring + **successor**, the **token race** (constant-size blake2b chain accumulator, Ed25519 receipts, originator completes the lap), and the **Autobase selfie gallery** — the originator creates a **per-wave** Autobase (namespace `wave-gallery:<waveId>`; key shared via gossip + token), peers request writer admission (`add-writer` = anti-spam gate) and `postSelfie()` a `wave-selfie` entry (inline JPEG thumbnail), all converging on one ordered gallery. Because the namespace is keyed by the random `waveId`, each wave/run starts empty (no stale selfies). The whole `storageDir/hyperwave` store is **wiped on startup** (galleries are ephemeral per-run — nothing there persists across runs), so disk doesn't accumulate; a persistent "past waves" browser is a future option. **Wave lifecycle (idle → lobby → racing → idle):** "kick off" **announces** a wave and opens a **lobby** (default 15s, `wave-announce`) so peers **opt in** (`wave-join`); the initiator then broadcasts `wave-start` (with the roster) and the token races. Exactly one wave at a time (anyone can start when idle; simultaneous starts resolve by lower-`waveId`-wins). **Everyone relays the ball (full-ring visual), but only opted-in roster members selfie** (`holding.canSelfie`); non-joiners just pass it on. Originator broadcasts `wave-end` on completion so all peers finish together; a timeout falls back to idle. **Join-time sync:** existing peers push a `wave-sync` (phase/roster/gallery key/lobby-time-left) to any newcomer on connect, so a peer joining mid-lobby/mid-race gets the right UI (join or spectate) and can't start a competing wave (broadcasts they missed wouldn't reach them). **Healing:** forward to the next _reachable_ peer clockwise; if the wave doesn't advance past my hop within `HEAL_TIMEOUT_MS` (the successor's `wave-pos` = ACK), skip that peer and re-forward. `seen` cleared per wave; `endedWaves` prevents reviving a finished wave. Renderer: ring, a **⚽ football token that rolls clockwise around the ring on every screen** (each holder broadcasts `wave-pos`; all renderers animate the ball → `position` event), Kick-off button, **lobby selfie capture** (camera + countdown during the lobby; the frame is staged and posted when the ball reaches the peer), and the gallery played **one selfie at a time in the ring centre** (newest arrival features as the wave passes, auto-cycles when idle). **WDK payment layer (complete):** `workers/lib/pay.js` = self-custodial Tron (Nile testnet) wallet — WDK is ESM-only so this CJS module bridges via dynamic `import()`; real transfers confirmed on-chain under Bare. **MVP payment currency = native TRX** (dropped TRC-20 USDT: no token contract, and a TRX transfer pays its own tiny fee from the same balance, so a wallet that received TRX can immediately send). **Money model = burned fees + gallery tips; NO sponsor rewards** (deliberately simplified — the interlocked payout, `wave-proof` receipt collection, golden-rule chain-walk, and gallery `burn-proof` op were all removed). Done: wallets (worker emits `wallet` {address,trx} IPC → renderer 💰 chip), gallery tipping (`wave-selfie` carries poster `address`; 💵 Tip button → `tip` IPC → `pay.send` → `tip-result` toast — **the only way to make money**), **participation fees burned** (1 TRX to Tron's black hole `T9yD14Nj9j7xAB4dbGeiX9h8unkKHxuWwb` = the unspendable all-zero EVM address; both the initiator on `start-wave` (`reason:'kickoff'`) and each joiner on `join-wave` (`reason:'join'`) — fire-and-forget, `burn-result` toast; skin in the game with no beneficiary, so no ledger/refunds. Zero-amount transfers are rejected by Tron, so a burn is a real small transfer; on-chain memo `hyperwave:<waveId>:<peerId>`), and a **paid-wave anti-spam gate** (the initiator's wave-announce is deferred until its kick-off burn is proven on-chain, carried as the signed `paid` attestation; peers ignore unpaid announces and won't `join()` until they verify the kick-off burn on-chain — `verifyBurnTx` — enforced only when a wallet is present; `wave-start`/`wave-sync` are gated on the same proof). An **OPTIMISTIC gallery admission** (`add-writer` carries the signed join attestation in `burn`; the admitter checks only the _signature_ — `burnAuthorizes` — with **NO on-chain call on the write path**, because that was O(N) REST reads concentrated on the admitter and didn't scale). The burn is verified where it has value: by **tippers/auditors** via the entry's `burnTx`. Spam is bounded deterministically in `apply()`: **one entry per peer + a byte-size cap** (`MAX_IMAGE_BYTES` 256KB / `MAX_CAPTION_BYTES` 512, `gallery.js`; oversized entries dropped). Trade-off: the old hard "no unpaid write, ever" gate becomes a **soft, publicly-detectable** one (a fake-burn seat is auditable). Verified live on Nile. **There are NO peer roles** — every peer runs the same code and behaves equally; there is no `role` option, no `HYPERWAVE_ROLE` env var, no `role` field in gossip, and no well-connected "seed" that others pin. The only asymmetry is **per-wave and belongs to that wave's initiator**: the peer that kicks off a wave keeps THAT wave's gallery Autobase open and retains it (so it survives for latecomers / replication — it is the archivist for its own wave only). If the initiator goes offline, its wave's gallery is not archived by anyone else — the accepted simplification of dropping the old standalone seed. Wallets must be **faucet-funded** (nileex gives TRX) to send. Fee flow shared by both engine hosts via `workers/lib/fees.js`. Adversarial hardening (see `docs/protocol.md` §11.2): gossip identity binding (self-describing fields must match the Noise connection id), signed `wave-end` completion + receipt-carrying stalls, paid-gate on every adoption path, heal-ACK precision, cheap-checks-before-verify. `spike/wdk/` + `spike/multiwriter/` = local de-risking spikes (gitignored). Design docs in `docs/` (`docs/idea.md` = the plain-language description of the built system).

**Module format:** Bare workers (`workers/`) are **CJS** (`require`/`module.exports`) — idiomatic for Bare/the template, and the worker entry is loaded by `PearRuntime.run`. The **renderer** (`renderer/`) is **ESM** (`import`/`export`, works over `file://`). This mix is intentional; Bare _can_ run ESM (`.mjs`), but converting the workers is all-or-nothing across the require/import boundary (`require()` of ESM throws) and the ESM worker-entry boot under pear-runtime is unverified — so we keep workers CJS.

Note: the gallery is gated at several points. (1) **admission** (`add-writer`, in `wave.js` `admitWriter`) is **OPTIMISTIC** when a wallet is present: the admitter checks only `burnAuthorizes` (the attestation _signature_, bound to peerId+wave) — **no on-chain call on the write path** (that was O(N) reads on the admitter). The burn is verified where it has value: tippers/auditors via `burnTx`. Spam is bounded by one-entry-per-peer + a byte cap. Admission **floods** the `add-writer` (relayed + dedup, re-stamping a fresh `mid` every `ADMIT_RETRY_MS` until `ADMIT_TIMEOUT_MS`) so it reliably reaches a current writer even across a sparse/churned mesh (a one-hop broadcast silently failed to admit survivors when peers died mid-wave and the requester wasn't directly connected to a writer — the flaky heal-test failure); it's authenticated by the carried receipt signature (`verifyReceipt` binds it to `peerId`), not the connection, so relaying stays sound. (2) `apply()` (in `gallery.js`) is deterministic (runs identically on every peer, no network I/O): appends a `wave-selfie` only if `receiptSig` verifies by `peerId`, within the **byte caps** (`MAX_IMAGE_BYTES`/`MAX_CAPTION_BYTES`); **one entry per peer** (first write wins — bounds the log); and keeps the tip `address` only if the entry's `burn` attestation names that wallet (`tronAddress === address`) so **every tip goes to the wallet that paid in**. (3) The gallery **key is signed** by the originator (`signGalleryKey` over `(waveId, autobaseKey)`, carried on wave-start/token/wave-sync as `keySig`/`autobaseKeySig`); peers `verifyGalleryKey` before opening, so a relay can't swap the key to a rogue Autobase. With no wallet (tests/headless) admission falls back to receipt-only and addresses are empty. Gotcha learned: capture `myBurnProof` into a local **before** the admission `await` in `postSelfie` — a fast wave can end mid-await and `resetSelfie` would blank it, stripping your own tip address.

**Gotchas learned:** (1) `package.json#upgrade` must be a valid `pear://` link (mint via `pear touch`) or `electron-forge start` refuses to boot. (2) Cold discovery on a fresh public-DHT topic takes **~20–35s** — launch demo instances early; consider a well-known bootstrap peer's `joinPeer` key or a local DHT bootstrap to speed it up. (3) Two Hyperswarm instances in one process don't reliably discover each other; always one instance per process. (4) A local DHT (`@hyperswarm/testnet` / `bootstrap` option) gives instant same-machine discovery and is confirmed working on the dev Mac (it only failed inside the build sandbox due to blocked loopback UDP) — use it for the live demo to avoid the ~30s public-DHT wait. (5) All instances on the same `matchId` share one ring; the app default (`hyperwave:demo-match:v1`) is fixed, so instances on the public DHT collide — isolate tests with `HYPERWAVE_MATCH`/the `matchId` engine option or the local bootstrap. Per-match topics are the production model.

## What This Project Is

A submission for the **Tether Developers Cup** (DoraHacks), theme "football / global tournament moment." It turns a stadium "Mexican wave" into a permissionless P2P relay:

- Peers join a match-specific Hyperswarm topic. Each peer's Noise public key deterministically maps to a fixed **position on the DHT ring** (angle derived from the key's top bytes) — the ring _is_ the stadium seating chart. No registration, no server.
- A signed **wave token** races peer-to-peer to each peer's **successor** (next live peer clockwise), accumulating a constant-size cryptographic receipt chain. Dead successors are skipped (self-healing).
- Selfies are captured **during the lobby** and posted to a shared per-wave **Autobase** gallery as the token reaches each participant (writes gated on a valid token receipt + burn attestation).
- **WDK** powers all money: self-custodial wallets, burned participation fees, and gallery tipping.

## Core Design Rules (baked-in decisions — do not re-litigate)

- **Money model = burned fees + tips; NO sponsor rewards.** The interlocked payout, validator role, and join bond were removed — sybil-farmable. Fees are burned to Tron's black hole (nobody profits); tips are the only earnings.
- **No peer roles.** Every peer runs the same code; the only asymmetry is per-wave (the initiator archives its own wave's gallery).
- **Constant-size hot path.** The token must NOT carry a growing `hops[]` array. Rolling accumulator: `newChainHash = blake2b(prevChainHash + thisReceiptSig)`.
- **One wire encoding (JSON).** Every gossip message AND the token are `JSON.stringify`'d over a single `compact-encoding` **`string`** Protomux frame (no binary token structs). Ed25519-signed over Noise-encrypted Hyperswarm streams.
- **Selfies never block the token.** Capture happens in the lobby (before the race); the staged frame posts when the ball arrives. The per-hop dwell is **experiential pacing, not security**.
- **Optimistic gallery admission.** The admitter checks only the burn attestation _signature_ — no on-chain call on the write path; the burn is verified where it pays off (tippers/auditors via `burnTx`).
- **Testnet only.** Native TRX on Nile (not TRC-20 USDT — a TRX transfer pays its own fee from the same balance). No smart contracts.

## Platform

Two hosts over one engine (`packages/hyperwave-lib-core`):

- **Desktop (primary): Electron forked from [`holepunchto/hello-pear-electron`](https://github.com/holepunchto/hello-pear-electron)** — run N peers on one laptop via `--storage <dir>`, reliable webcam via `getUserMedia`. Three-process split (respect it): **renderer** (Chromium, UI only, never touches the swarm), **Electron main** (window/lifecycle/IPC relay), **Bare worker** (all Holepunch P2P + WDK; a thin host over the engine's `init()`).
- **Mobile: Expo + react-native-bare-kit** (`apps/mobile/`) — same engine as a Bare worklet, same JSON IPC surface. Runs on the iOS simulator; rich UI/camera still open.
- **Spectator is just a peer instance.** The webcam selfie is the proof-of-humanity (no sensor layers).

## Stack

- **App shell:** Electron via hello-pear-electron (Electron Forge; `pear-runtime` for OTA updates).
- **Networking:** Hyperswarm (DHT discovery, NAT hole-punching, Noise duplex streams) — in the Bare worker.
- **Shared state:** Autobase (multi-writer selfie gallery; images inline as dataURLs — no Hyperblobs) + Corestore, in the worker.
- **Payments:** WDK — self-custodial, **native TRX on Tron Nile**, plain transfers (no contracts); runs **in the Bare worker** (WDK supports Bare). Packages: `@tetherto/wdk`, `@tetherto/wdk-wallet-tron`. WDK is ESM-only → `pay.js` bridges via dynamic `import()`.

## Message Types

As built (`docs/protocol.md` §5 has the schemas): `pointers` (heartbeat/liveness/Chord pointers), `wave-announce`, `wave-join`, `wave-start`, `wave-end` (flooded lifecycle), `token` (the racing ball, unicast), `wave-pos` (one-hop position/heal-ACK), `wave-sync` (join-time catch-up, unicast), `add-writer` (gallery admission, flooded), `find-succ`/`find-succ-reply` (distributed successor lookup). Gallery entries are `wave-selfie` ops inside the Autobase (not on-wire messages).
