# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Current State

The MVP is feature-complete (Electron forked from hello-pear-electron; the reusable engine is `packages/hyperwave-engine`). The on-wire protocol, crypto envelope, wave lifecycle (idle → lobby → racing → idle), the deterministic sweep, the multicore CRDT gallery, and the adversarial hardening are specified in `packages/hyperwave-engine/docs/protocol.md` (§11 = the hardening catalogue); the product in plain language is `docs/idea.md`; concurrent-waves/subscription scaling is in `scaling.md`. The money model (burned fees + tips, no rewards), the no-peer-roles rule, and the wallet abstraction are stated in Core Design Rules / Stack below — this section is the load-bearing facts and gotchas NOT captured in those:

- **Store wiped on startup.** The whole `storageDir/hyperwave` Corestore is erased at boot — galleries are ephemeral per-run (the feed namespace is keyed by the random `waveId`, so every run starts empty), nothing persists across runs, and a "past waves" browser is a future option. The exception: Cashu proofs are bearer funds held in `proof-store.js` **outside** the wiped store (Cashu is stateful).
- **Concurrent waves + subscription layer** (`scaling.md`). The wave FSM is multiplexed (`Map<waveId, WaveState>`) — several waves run at once on one topic; the old one-wave-at-a-time rule and lower-`waveId` tie-break are gone (`shouldAdopt` → `canAdopt`). Awareness of a wave opens no cores; `subscribe()`/`unsubscribe()` gate its feed cores (budget = O(subscribed)); `join()` implies subscribe; `autoSubscribe` (default true) auto-subscribes for the single-wave demo. Control gossip (join/start/sync) is forwarded only to a wave's subscribers (advertised via `subs`); only the tiny wave-announce floods the directory. Host commands stay backwards-compatible (`join(waveId?)`/`stageEntry({waveId?})` default to the newest joinable/joined wave).
- **Test vs mainnet is decided by the mint in the burn proof, NOT `walletType`** (which stays the generic `'cashu'`): `CashuWallet.verifyBurnTx` classifies the burn mint's network (`mint-networks.js`) against the local wallet's; a cross-network burn is a **definitive offline `wrong-network` rejection**, so peers drop cross-network waves from joins AND (renderer-side, `wallet-meta.js` `networkMatches`) from the directory/tips, re-filtered on a live mint switch. Unknown/custom mints are permissive. Any two mints on the same network interoperate. Detail: `docs/cashu.md`. **A second, coarser layer sits in front of this:** the DIRECTORY TOPIC is split by network so real-money and test-money peers never even discover each other. The engine stays network-agnostic — it exposes a generic live topic switch (`wave.setTopicId` / the `set-topic` command; leaves the old topic, drops connections, frees held waves, re-joins under the same swarm identity → ring seat stable). The network→topic **policy is the host's** (`workers/hyperwave.js` `topicForNetwork`): testnet/unknown/wallet-less on the base topic (unchanged), mainnet on `<base>:mainnet`. The worker taps the engine's `wallet` events and issues `set-topic` on a network change.
- **Tron black-hole address = `T9yD14Nj9j7xAB4dbGeiX9h8unkKHxuWwb`** (the unspendable all-zero EVM address); burns carry on-chain memo `hyperwave:<waveId>:<peerId>` and must be a real small transfer (Tron rejects zero-amount). Tron wallets must be **faucet-funded** (nileex) to send. The Cashu equivalent is a NUMS burn pubkey (`nums.js`).
- **The paid gate is soft and publicly-detectable** (a fake-burn seat is auditable), NOT hard — and there is **NO on-chain call on the ingest path** (O(N) REST reads didn't scale): every peer verifies only signatures (`verifyJoin`, plus `burnAuthorizes` when enforcing) then downloads block 0 only; the burn is re-checked where it pays off, by tippers/auditors via the entry's `burnTx`. A joiner whose burn confirms mid-lobby re-floods its join to attach it. Nile re-verification of the CRDT flow is pending (covered by the on-chain e2e tier).

**Everything runs under Bare** (the worker's real runtime), not Node — Node is only Electron's main process. Bare specifics: `bare-fs`, `bare-env` for env vars, `Bare.argv`/`Bare.exit`, global timers. Tests use **brittle** (Holepunch TAP framework), run under Bare: **`npm test`** (from repo root) runs **all three packages** — `hyperwave-engine` + `hyperwave-wallet-tron` + `hyperwave-wallet-cashu` — each `bare test.js` requiring its suites so brittle aggregates them (TAP, non-zero on failure). The wallet suites live in their respective wallet packages, the abstract `wallet` interface suite in `hyperwave-wallet`, the protocol suites in the engine. `npm run test:e2e:local` runs the 8-peer e2e harness; `npm run format` / `npm run lint` (from root) for prettier+lunte. Engine-internal dev commands (single-suite runs, the headless `wave.run.js`/`dht-local.js` smoke test) are in `packages/hyperwave-engine/CLAUDE.md`.

**Module format:** Bare workers (`workers/`) are **CJS** (`require`/`module.exports`) — idiomatic for Bare/the template, and the worker entry is loaded by `PearRuntime.run`. The **renderer** (`renderer/`) is **ESM** (`import`/`export`, works over `file://`). This mix is intentional; Bare _can_ run ESM (`.mjs`), but converting the workers is all-or-nothing across the require/import boundary (`require()` of ESM throws) and the ESM worker-entry boot under pear-runtime is unverified — so we keep workers CJS.

- `packages/hyperwave-engine/` — the reusable, **payment-agnostic** Bare engine (`createEngine`, the wave protocol, the CRDT feed; zero payment deps). Its `lib/` module map + dev workflow are in `packages/hyperwave-engine/CLAUDE.md` (loads only when you work under that dir); the authoritative spec is its `docs/protocol.md`.
- **The payment abstraction is split into its own packages** so a consumer installs neither the interface it doesn't need nor WDK/cashu-ts: `packages/hyperwave-wallet/` = the abstract `Wallet` base class (no deps — `require('hyperwave-engine').Wallet` is GONE, get it from `hyperwave-wallet`), `packages/hyperwave-wallet-tron/` (`TronWallet` native TRX + `TronUsdtWallet` USDT/TRC-20, WDK), `packages/hyperwave-wallet-cashu/` (`CashuWallet` ecash, type `'cashu'`, unit `sat`). Each concrete wallet depends on `hyperwave-wallet`; a host injects one via `createEngine({ deps: { createPayments } })`.
- The repo root = the Electron shell (`electron/`, `renderer/`, `workers/`). `workers/hyperwave.js` = the desktop host that injects `createCashuWallet` (**Cashu is the desktop default**; the engine ships no wallet). It is **init-message-driven** (built on main's `init` command) so wallet + swarm **seeds ride the IPC pipe, not argv/env**. Rationale worth knowing:
  - **Secure seed store** (`electron/main.js`): `safeStorage` OS keychain encrypts both seeds at `<storage>/{wallet,swarm}.seed.enc`; main mints the mnemonic + swarm seed itself (no engine change); Linux-without-keyring falls back to plaintext with a warning. See `docs/secure-seed-storage.md`.
  - **NSFW filter** (`renderer/lib/nsfw.js`): a per-peer viewing filter (the CRDT entry stays; hiding is local) via an esbuild bundle that embeds tfjs + nsfwjs + the model **in-memory** — no fetch, because a sandboxed `file://` renderer blocks fetch of local files. Lazy-loaded, fail-open.
  - **Wallet modal** (`renderer/lib/wallet.js`): the Tron account dropdown becomes a **mint picker** for Cashu (`set-wallet-options {mint}`), the faucet becomes **Top up** (`fund-wallet`), and chain-only UI (Tronscan/on-chain history/T-address send) is hidden. Labels use the wallet's real `unit` via `wallet-meta.js`. A Cashu tip rides `wave-note` (P2PK-locked token); the recipient sends `redeem {token}`. Engine commands: `set-wallet-options`, `fund-wallet`, `redeem` (currency-agnostic).
  - Ring angle is always derived locally from a peer's id (`angleOfId`), **never trusted from gossip**.
- Orientation pointers: `TODO.md` = the refinement backlog (check/update when picking up or finishing work); `DEMO.md` = local demo setup; `presentation/` = the slide deck (`generate_deck.py`). **Docs are distributed by concern**: the engine documents itself under `packages/hyperwave-engine/docs/` (`protocol.md` = authoritative spec, `usage.md` = API examples); the apps under `docs/` (`idea.md`, `hosting.md` = the Electron 3-process split + IPC seam, `secure-seed-storage.md`). Gotcha learned (gallery/feed): capture the burn proof + join sig into locals **before** the `postEntry` append `await` (`feed-crdt.js`) — a fast wave can end mid-append and a new wave's `clearBurnProof` would blank them, stripping your own tip address (the proof deliberately survives the end-of-wave reset; see `entry.js`). The gating mechanics (per-peer no-admission ingest, the pure deterministic `mergeFeed()` fold, no shared gallery key) are in `protocol.md` + `feed.js`/`feed-crdt.js`.

**Monorepo layout (npm workspaces).** `npm install` runs **once at the root** (hoists deps + links the workspace). Two root postinstall scripts matter: `fix-bare-engines` (normalizes deps' `engines` ranges bare-semver can't parse) and **`patch-secret-stream`** (vendors an opt-in `maxMessageSize` into `@hyperswarm/secret-stream` so the transport rejects an oversized message at its 3-byte length prefix, **before** its ~16 MB `allocUnsafe` — the engine sets it per connection on an owned swarm, default 1 MB); both run again in the desktop bundle via `forge.config.js`. The engine was extracted so the **SAME engine boots under Electron** (`workers/hyperwave.js`) **and a react-native-bare-kit worklet** (`worklet/app.js`) — both are ~20-line hosts over `createEngine`. **Historical path note:** older `app/workers/lib/X` references mean `packages/hyperwave-engine/lib/X`; `app/renderer|electron` → `renderer|electron` at the root.

**`mobile/`** = the mobile host (Expo + `react-native-bare-kit`, scaffold): `App.js` + `src/useEngine.js` boot the worklet bundle and speak the same JSON IPC protocol as the desktop renderer. Runs on the iOS simulator; the Bare native addons (41 xcframeworks) are linked via `scripts/link-ios-addons.mjs` (`bare-link` from `mobile/`, because react-native-bare-kit's prepare_command scans the barren monorepo root + CocoaPods skips local pods; wired into `postinstall` + `npm run ios`). Verified cross-peer with a desktop peer over the public DHT, WDK wallet runs in the worklet. Remaining: rich RN UI, camera capture, secure/funded seed, Android addons (`mobile/README.md`).

**Gotchas learned:** (1) `package.json#upgrade` must be a valid `pear://` link (mint via `pear touch`) or `electron-forge start` refuses to boot. (2) Cold discovery on a fresh public-DHT topic takes **~20–35s** — launch demo instances early; consider a well-known bootstrap peer's `joinPeer` key or a local DHT bootstrap to speed it up. (3) Two Hyperswarm instances in one process don't reliably discover each other; always one instance per process — so a host that ALSO uses Hyperswarm should pass its instance to `createEngine({ swarm })` / `createWave({ swarm })` (the engine shares it and never destroys it; on close it only leaves the topics it joined + detaches its listeners) rather than let the engine open a second. (4) A local DHT (`@hyperswarm/testnet` / `bootstrap` option) gives instant same-machine discovery and is confirmed working on the dev Mac (it only failed inside the build sandbox due to blocked loopback UDP) — use it for the live demo to avoid the ~30s public-DHT wait. (5) All instances on the same `topicId` share one ring; the app default (`hyperwave:demo:v1`) is fixed, so instances on the public DHT collide — isolate tests with `HYPERWAVE_TOPIC`/the `topicId` engine option or the local bootstrap. Per-room topics are the production model.

## What This Project Is

A submission for the **Tether Developers Cup** (DoraHacks). The cup's brief — a _global tournament moment_ — is realized here as a worldwide **wave of moments**: peers around the world capture a moment together as a bitcoin-orange wave (⚡ motif) sweeps the ring. A permissionless P2P relay:

- Peers join a room-specific Hyperswarm topic. Each peer's Noise public key deterministically maps to a fixed **position on the DHT ring** (angle derived from the key's top bytes) — the ring _is_ the map of participants around the world. No registration, no server.
- A **deterministic sweep** rolls the wave around that ring: the initiator floods a start time, and every joined peer fires at its own angle-ordered moment — like a ripple around a circle, choreography rather than a passed object. The wave takes a fixed few seconds no matter how many peers; a dead peer's moment simply passes.
- Moments are captured **during the lobby** and posted, when each peer's sweep slot fires, to a per-wave **multicore CRDT** gallery — one Hypercore per participant, merged locally into one ordered view (no indexer, no admission; writes gated on a signed join attestation + burn attestation).
- **The money is bitcoin:** the desktop's self-custodial wallet is **Cashu** — Chaumian ecash on a Bitcoin **Lightning** mint (unit `sat`, the ⚡ motif) — powering burned participation fees and gallery tips. A Tron wallet (native TRX / USDT via **WDK**) is a pluggable alternative.

## Core Design Rules (baked-in decisions — do not re-litigate)

- **The engine (`packages/hyperwave-engine`) is theme-agnostic; keep it that way.** It speaks generic concepts only — a **wave** that **sweeps** a **ring**, an **entry** with an **opaque `payload`** in a per-wave **feed**, a cosmetic per-peer **tag**, a shared **topic** (`topicId`). The app's "moments wave" (moments/selfies, gallery, countries, the ⚡ spark, wave-start) lives entirely in the desktop app (repo root) / `mobile/`, which map their concepts onto the engine's at the IPC boundary (`renderer/lib/ipc.js` + `app.js`: a moment `{image,caption}` → the entry payload; a country → the tag; `set-tag`/`stage-entry`/`feed` on the wire). Do NOT reintroduce app/theme vocabulary (moment/selfie/gallery/country/kickoff/stadium/football) into the engine package.
- **Money model = burned fees + tips; NO sponsor rewards.** Payouts/rewards of any kind are sybil-farmable — do not add them. Fees are burned to Tron's black hole (nobody profits); tips are the only earnings.
- **No peer roles.** Every peer runs the same code; the only asymmetry is per-wave (the initiator archives its own wave's gallery).
- **The wave is a deterministic sweep, not a passed object.** `wave-start` carries `t0`+`lapMs`; every roster peer self-triggers at its angle-ordered slot (`sweep.js`) and self-completes at `t0+lapMs+grace`. No token, no healing, no completion message — wall-clock is a chosen constant regardless of N, and a dead peer's slot simply passes.
- **One wire encoding (JSON).** Every gossip message is `JSON.stringify`'d over a single `compact-encoding` **`string`** Protomux frame, and carries the uniform Ed25519 envelope (`origin`/`ts`/`sig`, protocol.md §5.0) verified at the receive edge — over Noise-encrypted Hyperswarm streams. Domain attestations (join/burn) sign their own tuples underneath.
- **Moments never block the wave.** Capture happens in the lobby (before the sweep); the staged frame posts when the peer's slot fires. Any experiential pacing is a renderer concern, not the protocol.
- **Multicore CRDT gallery — no admission, no indexer.** Each participant owns one Hypercore and posts its single moment at block 0; its key rides its own `wave-join` (self-certified by the join attestation). Every peer ingests every join, opens that core, downloads block 0 only, and merges locally (`mergeFeed`) — same cores in, byte-identical gallery out. The gate is signature-only (`verifyJoin`, plus `burnAuthorizes` when paid) with no on-chain call on the ingest path; the burn is verified where it pays off (tippers/auditors via `burnTx`). Spam is bounded by block-0-only download → one entry per peer + byte caps. A join that misses the lobby spectates.
- **Testnet only.** The DESKTOP default is **Cashu** (testnet ecash on a Lightning mint; the dev default is the free auto-paying `testnut` mint). The Tron wallet uses native TRX on Nile (not TRC-20 USDT — a TRX transfer pays its own fee from the same balance). No smart contracts.

## Platform

Two hosts over one engine (`packages/hyperwave-engine`):

- **Desktop (primary): Electron forked from [`holepunchto/hello-pear-electron`](https://github.com/holepunchto/hello-pear-electron)** — run N peers on one laptop via `--storage <dir>`, reliable webcam via `getUserMedia`. Three-process split (respect it): **renderer** (Chromium, UI only, never touches the swarm), **Electron main** (window/lifecycle/IPC relay), **Bare worker** (all Holepunch P2P + the payment wallet — Cashu by default; a thin host over the engine's `createEngine()`).
- **Mobile: Expo + react-native-bare-kit** (`mobile/`) — same engine as a Bare worklet, same JSON IPC surface. Runs on the iOS simulator; rich UI/camera still open.
- **Spectator is just a peer instance.** The webcam moment is the proof-of-humanity (no sensor layers).

## Stack

- **App shell:** Electron via hello-pear-electron (Electron Forge; `pear-runtime` for OTA updates).
- **Networking:** Hyperswarm (DHT discovery, NAT hole-punching, Noise duplex streams) — in the Bare worker.
- **Shared state:** a multicore CRDT moment gallery — one Hypercore per participant (merged locally, no Autobase/indexer; images inline as dataURLs — no Hyperblobs) over Corestore, in the worker.
- **Payments:** the engine talks to payments through the abstract **`Wallet` base class** (`hyperwave-wallet/lib/wallet.js`) — the concrete interface (`type`/`fee`/`address`/`balances`/`send`/`burn`/`verifyBurnTx`/`transactions`/`dispose`); an app plugs in its own by injecting a factory returning a `Wallet` subclass (`createEngine({ deps: { createPayments } })`). The **desktop default** is **`CashuWallet`** (`hyperwave-wallet-cashu`, cashu-ts) — self-custodial **Chaumian ecash on a Bitcoin Lightning mint** (unit `sat`; the free auto-paying `testnut` mint by default); it runs **in the Bare worker**. The pluggable alternative is **`TronWallet`** (`hyperwave-wallet-tron`) = WDK, native TRX on Tron Nile, plain transfers (no contracts); WDK is ESM-only → it bridges via dynamic `import()`. The wallet-agnostic fee flows live in `payments.js`. Each wallet declares a `type` (`'cashu'` / `'tron-nile'`) that rides paid waves' announces so a joiner only joins a wave whose payment mechanism it supports.

## Code Style (beyond prettier/lunte — apply to all new and edited code)

- **Always brace `if`/`else`/`for`/`while` bodies** — no brace-less single-statement bodies, even one-liners.
- **No single-letter / cryptic variable names** (`s`, `f`, `r`, `t`, `pt`, `it`, `k`…). Use descriptive names. Acceptable idioms: loop index `i`, geometry locals (`dx`/`dy`/`x`/`y`), `ctx` (canvas), `ev`/`evt` (events).
- **Avoid `switch` statements where possible** — prefer a lookup table (object keyed by case → value or handler function).
- **Avoid deeply nested `if`/`else`** — prefer early returns / guard clauses; extract a helper when branching gets deep. Same spirit applies to nested ternaries.
- **One statement per line** — never `a; b` on one line or comma-operator statement chaining.
- **Always use semicolons** — enforced by prettier (`.prettierrc.js` overrides holepunch's `semi: false` with `semi: true`).
- **80-character line limit** — enforced by prettier (`.prettierrc.js` overrides holepunch's `printWidth: 100` with `printWidth: 80`).
- **At most two chained calls per line** — `funcA().funcB()` is fine; `funcA().funcB().funcC()` is not. For 3+ calls, each call goes on its own indented line under the receiver. Caveat: prettier only produces that broken form when the chain exceeds 80 chars — it collapses a manually broken short chain back onto one line (no config option controls this) — so for a 3+ chain that fits in 80 chars, split it with an intermediate variable instead (`const hex = BigInt(nid).toString(16); return hex.padStart(16, '0');`). Not prettier-enforceable; apply by convention.
- **Always strict equality** — `===`/`!==`, never `==`/`!=`.
- **Declare all variables at the top of their scope** — module state/DOM refs/config consts at the top of the module, function locals at the top of the function (guard clauses may precede them). Function and handler-table definitions are not "variables" and live in their sections.
- **Never `setInterval`** — use a self-rescheduling `setTimeout`: the callback does its work, then re-arms itself (`timer = setTimeout(tick, ms)`) as its **last** step. Ticks can never overlap/stack, a slow or async tick delays the next instead of racing it, and each tick can decide not to continue. Clear with `clearTimeout`; the handle changes every tick, so keep it in a `let` and re-check it after async work before re-arming.
- **Google TypeScript Style Guide** (https://google.github.io/styleguide/tsguide.html) is the baseline for anything not covered above, where it applies to plain JS and doesn't conflict with this list or the prettier config. Notably: named exports only (no `export default`; exception: framework-required, e.g. the Expo root component), `const`/`let` never `var`, `CONSTANT_CASE` for module-level constants, `camelCase`/`PascalCase` otherwise, prefer `for…of`.
- **No functions with more than 3 positional parameters** — take a single destructured options object instead (`function myFunc({ waveId, hopCount, chainHash, sigHex })`; defaults inline, as in `finishWave(waveId, { stalled = false } = {})`). The real hazard is adjacent same-typed args (hex strings especially) silently transposed at a call site, so prefer the object form even at 2–3 args when the types are interchangeable. **Enforced in `packages/hyperwave-engine/`; advisory elsewhere.** Exemption: geometry/canvas draw helpers (renderer), where positional `(x, y, r, …)` is the universal idiom.
- **No pass-through `return func()` unless the return value is meant** — only write `return func()` when the caller's contract is to return `func()`'s value/type. If the call is done just for its effect (or the function returns nothing meaningful), call it on its own line and `return;` separately, so the function doesn't accidentally leak a return value it never promised.
- **Class private members use `#`, not `_`** — real ES private fields/methods (`#waiters`, `#ingest()`), the modern way, enforced by the language rather than convention. (An `_` prefix remains fine for its _other_ job: marking deliberately-unused parameters, e.g. `(_evt, url) =>`. Third-party internals like WDK's `_tronWeb` are theirs to name.)
