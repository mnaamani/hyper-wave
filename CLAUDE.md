## Current State

The MVP is feature-complete. `app/` is the Electron app (forked from hello-pear-electron). Build-order Steps 1–4 done: discovery, **pointers-heartbeat gossip** (Protomux channel; a single `pointers` message carries liveness + country + role + Chord succ/pred), live sorted ring + **successor**, the **token race** (constant-size blake2b chain accumulator, Ed25519 receipts, originator completes the lap), and the **Autobase selfie gallery** — the originator creates a **per-wave** Autobase (namespace `wave-gallery:<waveId>`; key shared via gossip + token), peers request writer admission (`add-writer` = anti-spam gate) and `postSelfie()` a `wave-selfie` entry (inline JPEG thumbnail), all converging on one ordered gallery. Because the namespace is keyed by the random `waveId`, each wave/run starts empty (no stale selfies). The whole `storageDir/hyperwave` store is **wiped on startup** (galleries are ephemeral per-run — nothing there persists across runs), so disk doesn't accumulate; a persistent "past waves" browser is a future option. **Wave lifecycle (idle → lobby → racing → idle):** "kick off" **announces** a wave and opens a **lobby** (default 15s, `wave-announce`) so peers **opt in** (`wave-join`); the initiator then broadcasts `wave-start` (with the roster) and the token races. Exactly one wave at a time (anyone can start when idle; simultaneous starts resolve by lower-`waveId`-wins). **Everyone relays the ball (full-ring visual), but only opted-in roster members selfie** (`holding.canSelfie`); non-joiners just pass it on. Originator broadcasts `wave-end` on completion so all peers finish together; a timeout falls back to idle. **Join-time sync:** existing peers push a `wave-sync` (phase/roster/gallery key/lobby-time-left) to any newcomer on connect, so a peer joining mid-lobby/mid-race gets the right UI (join or spectate) and can't start a competing wave (broadcasts they missed wouldn't reach them). **Healing:** forward to the next _reachable_ peer clockwise; if the wave doesn't advance past my hop within `HEAL_TIMEOUT_MS` (the successor's `wave-pos` = ACK), skip that peer and re-forward. `seen` cleared per wave; `endedWaves` prevents reviving a finished wave. Renderer: ring, a **⚽ football token that rolls clockwise around the ring on every screen** (each holder broadcasts `wave-pos`; all renderers animate the ball → `position` event), Kick-off button, **lobby selfie capture** (camera + countdown during the lobby; the frame is staged and posted when the ball reaches the peer), and the gallery played **one selfie at a time in the ring centre** (newest arrival features as the wave passes, auto-cycles when idle). **WDK payment layer (complete):** `workers/lib/pay.js` = self-custodial Tron (Nile testnet) wallet — WDK is ESM-only so this CJS module bridges via dynamic `import()`; real transfers confirmed on-chain under Bare. **MVP payment currency = native TRX** (dropped TRC-20 USDT: no token contract, and a TRX transfer pays its own tiny fee from the same balance, so a wallet that received TRX can immediately send). **Money model = burned fees + gallery tips; NO sponsor rewards** (deliberately simplified — the interlocked payout, `wave-proof` receipt collection, golden-rule chain-walk, and gallery `burn-proof` op were all removed). Done: wallets (worker emits `wallet` {address,trx} IPC → renderer 💰 chip), gallery tipping (`wave-selfie` carries poster `address`; 💵 Tip button → `tip` IPC → `pay.send` → `tip-result` toast — **the only way to make money**), **participation fees burned** (1 TRX to Tron's black hole `T9yD14Nj9j7xAB4dbGeiX9h8unkKHxuWwb` = the unspendable all-zero EVM address; both the initiator on `start-wave` (`reason:'kickoff'`) and each joiner on `join-wave` (`reason:'join'`) — fire-and-forget, `burn-result` toast; skin in the game with no beneficiary, so no ledger/refunds. Zero-amount transfers are rejected by Tron, so a burn is a real small transfer; on-chain memo `hyperwave:<waveId>:<peerId>`), and a **paid-wave anti-spam gate** (the initiator's wave-announce is deferred until its kick-off burn is proven on-chain, carried as the signed `paid` attestation; peers ignore unpaid announces and won't `join()` until they verify the kick-off burn on-chain — `verifyBurnTx` — enforced only when a wallet is present; `wave-start`/`wave-sync` are gated on the same proof). An **OPTIMISTIC gallery admission** (`add-writer` carries the signed join attestation in `burn`; the admitter checks only the _signature_ — `burnAuthorizes` — with **NO on-chain call on the write path**, because that was O(N) REST reads concentrated on the admitter and didn't scale). The burn is verified where it has value: at **raffle payout** (the seed verifies the _winner's_ burn on-chain, walking the ranking past any fake-burn entries) and by **tippers/auditors** via the entry's `burnTx`. Spam is bounded deterministically in `apply()`: **one entry per peer + a byte-size cap** (`MAX_IMAGE_BYTES` 256KB / `MAX_CAPTION_BYTES` 512, `gallery.js`; oversized entries dropped). Trade-off: the old hard "no unpaid write, ever" gate becomes a **soft, publicly-detectable** one (a fake-burn seat wins nothing + is auditable). Verified live on Nile. The **validator/seed** role is a gallery archivist (retains galleries so they survive peers leaving) and, optionally, the **raffle sponsor**. **Sponsor-funded raffle** (`workers/lib/wave.js` `runRaffle`, off unless a seed sets `raffleTrx`/`HYPERWAVE_RAFFLE_TRX>0`): a positive incentive re-added _without_ the sybil-farmable payout — after each wave the seed draws ONE winner among gallery participants and pays `raffleTrx` to the winner's burn-verified address (net for a participant: −fee +prize). Fairness = **commit-reveal, internal (no external beacon)**: each participant publishes `commit=H(secret)` **two ways** with the same value — ring-signed on `wave-join`/`wave-announce` gossip (`signCommit`/`verifyCommit`), which the seed caches **in memory** during its lobby (phase gate = reveal deadline), AND **on-chain in its fee-burn memo** (`hyperwave:<waveId>:<peerId>:<commit>`, timestamped, immutable); it reveals `secret` in its `wave-selfie` entry (which keeps `burnTx` so the on-chain commit is locatable). The seed **draws from the fast in-memory gossip cache — it does NOT fetch commits over the Tron REST API at draw time** (too slow / rate-limited); the on-chain memo is the **accountability record**: because commits (chain) + reveals (gallery) are both public, anyone can _later_ recompute the eligible set + winner and catch a seed that dropped a validly-committed participant (a deterrent, not seed-trust). Draw = `token.js` `raffleDraw` folds the reveals into a seed + a deterministic **ranking**; `runRaffle` walks the ranking and pays the FIRST candidate whose burn **verifies on-chain** (this is where optimistic admission's deferred burn check happens — O(1) reads for the winner, not O(N) at admission) (unit-tested; verified live on Nile — commit in the memo, seed drew, verified the winner's burn, and paid). `verifyBurnTx` returns the memo `commit` for verifier tooling; `apply()` keeps `burnTx`. **Trust simplification: sponsor = admitter = seed** (one trusted role funds + records commits + draws), so it could censor entries — production must **separate the admitter (independent originator) from the prize-holder** (noted in `docs/protocol.md` §12 + `ideas/raffle.md`). A paid-entry game of chance is legally a **lottery** → testnet-only. Gotcha (same as burns): capture `myRaffleSecret` into a local **before** the admission `await` in `postSelfie`, else a fast wave's `resetSelfie` blanks the reveal. Wallets must be **faucet-funded** (nileex gives TRX) to send. Fee flow shared by both engine hosts via `workers/lib/fees.js`. Adversarial hardening (see `docs/protocol.md` §11.2): gossip identity binding (self-describing fields must match the Noise connection id), signed `wave-end` completion + receipt-carrying stalls, paid-gate on every adoption path, heal-ACK precision, cheap-checks-before-verify. `spike/wdk/` + `spike/multiwriter/` = local de-risking spikes (gitignored). Design docs in `ideas/` (note: `ideas/final-idea.md` still describes the original interlocked-reward design — superseded by the fees+tips model above).

**Gotchas learned:** (1) `package.json#upgrade` must be a valid `pear://` link (mint via `pear touch`) or `electron-forge start` refuses to boot. (2) Cold discovery on a fresh public-DHT topic takes **~20–35s** — launch demo instances early; consider a validator `joinPeer` well-known key or local DHT bootstrap to speed it up. (3) Two Hyperswarm instances in one process don't reliably discover each other; always one instance per process. (5) All instances on the same `matchId` share one ring; the app default (`hyperwave:demo-match:v1`) is fixed, so instances on the public DHT collide — isolate tests with `HYPERWAVE_MATCH`/the `matchId` engine option or the local bootstrap. ## What This Project Is

A submission for the **Tether Developers Cup** (DoraHacks), theme "football / global tournament moment." It turns a stadium "Mexican wave" into a permissionless P2P relay:

- Peers join a match-specific Hyperswarm topic. Each peer's Noise public key deterministically maps to a fixed **position on the 256-bit DHT ring** (`angle = uint256(pubkey) / 2^256 * 360°`) — the ring _is_ the stadium seating chart. No registration, no server.
- A signed **wave token** races peer-to-peer to each peer's **successor** (next live peer clockwise), accumulating a cryptographic receipt chain (~50–100ms/hop). Dead successors are skipped using a gossip-built ring map.
- After the token passes, each peer gets a **proof window** to react, take a selfie, and write it to a shared **Autobase** gallery (writes gated on holding a valid token receipt).
- **WDK** powers all money: self-custodial wallets, a join bond, interlocked sponsor payouts, and gallery tipping.

## Core Design Rules (baked-in decisions — do not re-litigate)

These are refinements already settled in `final-idea.md`; honor them when implementing:

> **SUPERSEDED (money model):** sponsor rewards have been **removed** from the product. The three reward-related rules below — the interlocked/golden-rule payout, "peers push proofs to the validator", and "validator pays from a budget/ledger" — no longer apply. The money model is now **burned fees + gallery tips only**; the validator/seed is a pure gallery archivist. The constant-size accumulator is kept (it still backs the token mechanic + gallery write-gate), but nothing "reassembles the chain for payout". Keep this in mind when reading the reward-flavoured rules below.

- **Constant-size hot path.** The `wave-token` must NOT carry the full growing `hops[]` array (bloats to 100s of KB and blows the timing budget). Use a rolling accumulator: `newChainHash = blake2b(prevChainHash + thisReceiptSig)`. Each peer gossips its individual receipt to the validator, which reassembles the full chain off the hot path. Target ~200–400 bytes/token.
- **Two encodings by layer.** Race layer (`wave-token`) → Compact Encoding (binary). Gossip messages and Autobase writes → JSON. Everything is Ed25519-signed over Noise-encrypted Hyperswarm streams.
- **Write to Autobase during the proof window, not "after the race"** — peers close the app once the fun is over, so writes must happen while still connected. The Sponsor Validator also replicates the Autobase + Hyperblobs so the gallery survives after peers leave.
- **Interlocked reward (the golden rule):** peer N is paid only when proof shows peer N+1 continued the wave. On a break, pay the longest valid prefix.
- **Validator is a first-class swarm peer**, not an edge server. Peers push proofs _directly_ to it (via `swarm.joinPeer(validatorKey)`); gossip is a redundant path, never the critical path for reward delivery. Reward is fixed-per-participant (budgeted spend), never a split pot — a split pot creates censorship incentives and collapses.
- **MVP is trusted-validator + testnet.** WDK is a wallet SDK, not an escrow platform: bond = a plain testnet USDT transfer to the validator, tracked in the validator's internal ledger; payouts = a loop of individual transfers. No smart contracts in the MVP. Prefer **Tron testnet** (cheapest/fastest USDT).
- The 1-second anti-bot proof delay is **experiential pacing, not security** — do not present it as a real anti-bot measure.

## MVP Platform: Desktop Electron (decided — see `final-idea.md` §11)

The MVP is a **desktop Electron app forked from [`holepunchto/hello-pear-electron`](https://github.com/holepunchto/hello-pear-electron)**, NOT React Native/mobile. Reasons: run N peers on one laptop via `--storage <dir>` (no NAT roulette in the live demo), reliable webcam via `getUserMedia`, fast iteration.

Consequences that override the earlier mobile design:

- **No accelerometer.** The L2 Kinetic / sensor proof layers are **cut**. The **Autobase webcam selfie is the proof-of-humanity** (physical action + gallery content + humanity in one snap). Keep the compact token accumulator regardless.
- **Three-process split imposed by the template — respect it:**
  - **Renderer (Chromium):** UI only — ring viz, lobby selfie capture, gallery, tipping, wallet chip. Never touches the swarm; talks over the IPC bridge.
  - **Electron main (Node.js):** window/lifecycle, Pear update bridge, IPC routing.
  - **Bare worker(s):** all Holepunch P2P — Hyperswarm, token race, gossip, Autobase, Hyperblobs, Corestore — **plus WDK** (WDK officially supports Bare, so payments live alongside the swarm in one worker; renderer requests tips/payouts over IPC).
- **Validator and spectator are just peer instances** run with a role flag (validator gets a visible log panel; spectator joins `client-only`).

## Intended Stack

- **App shell:** Electron via hello-pear-electron (Electron Forge; `pear-runtime` for OTA updates).
- **Networking:** Hyperswarm (DHT discovery, NAT hole-punching, Noise duplex streams) — in the Bare worker.
