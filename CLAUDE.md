## Current State

The MVP is feature-complete. The Electron app lives at the repo root (forked from hello-pear-electron). Built: discovery, **heartbeat gossip** (Protomux channel; a single `heartbeat` message carries liveness + country — no ring structure), live sorted ring, the **deterministic sweep** (`wave-start` carries `t0` + `lapMs`; every roster peer derives the identical angle-ordered schedule from the canonical roster — pure `sweep.js` — and self-triggers at its own slot; the wave ends deterministically at `t0+lapMs+grace` on every peer, no completion message, no healing; a dead peer's slot just passes), and the **multicore CRDT selfie gallery** (`feed-crdt.js`) — each participant owns ONE Hypercore in the per-wave namespace `wave-feed:<waveId>` and appends its single `wave-entry` (inline JPEG thumbnail) at its sweep slot; its core key rides its `wave-join` (self-certified by the **join attestation** — `attest.js signJoin` over `waveId|peerId|writerKey`), so **every peer** ingests every `wave-join`, opens that core (`addWriter`), and merges the set locally (`mergeFeed`) — no indexer, no admission, no shared gallery key. wave-start/wave-sync carry the full `writers` set so a late adopter is self-contained. Because the namespace is keyed by the random `waveId`, each wave/run starts empty (no stale selfies). The whole `storageDir/hyperwave` store is **wiped on startup** (galleries are ephemeral per-run — nothing there persists across runs), so disk doesn't accumulate; a persistent "past waves" browser is a future option. **Wave lifecycle (idle → lobby → racing → idle):** "kick off" **announces** a wave and opens a **lobby** (default 15s, `wave-announce` — no gallery key, none exists) so peers **opt in** (`wave-join`, which publishes that peer's own gallery core: writer key + join attestation + burn); every peer ingests every join and opens that core, and at lobby close the initiator floods `wave-start` (the full `writers` set + `t0` + `lapMs`; the roster is derived as `{by} ∪ writers[].peerId`), and the sweep runs. **Concurrent waves** (scaling.md Phase 1): the wave FSM is multiplexed (`Map<waveId, WaveState>`) — several waves can be engaged at once on the one topic; the old "exactly one wave at a time" rule and lower-`waveId`-wins tie-break are gone (`shouldAdopt` → `canAdopt`). Each wave owns its own timers/`EntryPipeline`/roster/feed; the host command surface stays backwards-compatible (`join(waveId?)`/`stageEntry({waveId?})` default to the newest joinable/joined wave; the `feed` message gains an additive `waveId`). **Subscription layer** (scaling.md Phase 2/3): being AWARE of a wave (saw its announce) opens no cores; `subscribe()` opens its feed + `unsubscribe()` closes it (join/leave lifecycle; `join()` implies subscribe), bounding the core budget to O(subscribed). `autoSubscribe` (createWave option, default true) auto-subscribes on awareness for the single-wave demo UX; a host sets it false for true browse-then-pick (`subscribe-wave`/`unsubscribe-wave` IPC commands). Control gossip is **scoped**: a wave's join/start/sync are forwarded only to neighbours that advertised (via `subs`) they're subscribed to it — O(subscribed), not O(topic); the tiny wave-announce still floods the directory. Each subscribed wave also joins its own sub-topic `hash(prefix:topic:wave)` for participant discovery off the O(N) mesh. (Directory-at-scale indexing is Phase 4.) **Only opted-in roster members selfie**; everyone renders the sweep. Every peer self-completes at `t0+lapMs+END_GRACE_MS` — there is no `wave-end` message and nothing to heal; receiver-side clamps (lapMs cap, t0 horizon) stop a hostile start from wedging a wave open. `endedWaves` prevents reviving a finished wave id. **Join-time sync:** existing peers push a `wave-sync` (phase/`writers`/`t0`/`lapMs`/lobby-time-left) to any newcomer on connect, so a peer joining mid-lobby/mid-race gets the right UI (join or spectate) for each engaged wave (it is caught up on every one, not forced to pick). Renderer: ring, a **⚽ football that rolls clockwise around the ring on every screen** (each engine walks the shared schedule locally and emits `position` events — no gossip), Kick-off button, **lobby selfie capture** (camera + countdown during the lobby; the frame is staged and posted when the peer's slot fires), and the gallery played **one selfie at a time in the ring centre** (newest arrival features as the wave passes, auto-cycles when idle). **WDK payment layer (complete):** `workers/lib/wallet.js` = self-custodial Tron (Nile testnet) wallet — WDK is ESM-only so this CJS module bridges via dynamic `import()`; real transfers confirmed on-chain under Bare. **The DESKTOP now defaults to Cashu** — Chaumian ecash on a Lightning-connected mint (`cashu-wallet.js`; the worker injects `createCashuWallet` as the engine's payment factory). Same money model, no gossip-protocol change: the black-hole address becomes a NUMS pubkey (`nums.js`), the memo a mint-signed NUT-11 tag, `verifyBurnTx` a NUT-07 checkstate; `walletType` is the generic `'cashu'` (any LN Cashu peer interoperates, regardless of mint), unit `sat`. Zero-amount transfers are rejected by Tron, so a burn is a real small transfer; on-chain memo `hyperwave:<waveId>:<peerId>`), and a **paid-wave anti-spam gate** (the initiator's wave-announce is deferred until its kick-off burn is proven on-chain, carried as the signed `paid` attestation; peers ignore unpaid announces and won't `join()` until they verify the kick-off burn on-chain — `verifyBurnTx` — enforced only when a wallet is present; `wave-start`/`wave-sync` are gated on the same proof). **No admission — every peer ingests every join** (each `wave-join` carries the joiner's writer key + signed join attestation + burn attestation; every peer verifies the _signatures_ — `verifyJoin`, plus `burnAuthorizes` when enforcing — then opens that participant's core and downloads block 0 only, with **NO on-chain call on the ingest path**, because O(N) REST reads didn't scale; a joiner whose burn confirms mid-lobby re-floods its join to attach it). Spam is bounded deterministically in `mergeFeed()`: block-0-only download → **one entry per peer + a byte-size cap** (`MAX_PAYLOAD_BYTES` 256KB, `feed.js`; oversized entries dropped). Trade-off: the paid gate is **soft and publicly-detectable** (a fake-burn seat is auditable) rather than hard. The initiator is an ordinary participant (no indexer, no admission, no retention role); galleries are ephemeral (a subscribed wave's cores stay open — the post-race idle gallery + latecomer replication rely on it — until `unsubscribe()` (join/leave lifecycle, scaling.md Phase 2) or `close()`; nothing persists across runs since the store is wiped on startup — no "past waves" feature). Feeds are NOT auto-closed at wave-end (final-slot entries keep replicating past the deterministic end, so closing there would race convergence). Wallets must be **faucet-funded** (nileex gives TRX) to send. Fee flow shared by both engine hosts via the same `workers/lib/wallet.js`. Adversarial hardening (see `packages/hyperwave-engine/docs/protocol.md` §11): gossip identity binding (self-describing fields must match the Noise connection id), paid-gate on every adoption path (now enforced per-peer on each `wave-join`), self-certifying gallery cores — each participant's core key is bound by its join attestation (a relay can't forge/substitute a core key under another peerId, and there's no shared gallery key to attack), hostile-start clamps (lapMs cap / t0 horizon), deterministic end (completion needs no trust). `spike/wdk/` + `spike/multiwriter/` = local de-risking spikes (gitignored). Note: the gallery is gated at several points. (1) **Ingest — every peer, no admission** (`feed-crdt.js` `addWriter`, driven by `wave.js` `ingestWriter` on each `wave-join`/`writers` entry): each flooded `wave-join` carries the participant's own gallery-core key + join attestation + (when paid) burn attestation; **every** peer checks only the _signatures_ (`verifyJoin` — which covers the writer key, so a relay can't forge/substitute a core key under another peerId — plus `burnAuthorizes` when enforcing, per-peer on a direct join) then opens that core and downloads block 0 only — **no on-chain call on the ingest path**. A participant posts to **its own** core (`postEntry` appends at block 0), so there is no writable-wait. A joiner whose burn confirms mid-lobby re-floods its join to attach it. (2) `mergeFeed()` (in `feed.js`) is a pure deterministic fold (runs identically on every peer, no network I/O): keeps a `wave-entry` only if its **join attestation** verifies (`verifyJoin` by `peerId` over waveId+writerKey), within the **byte caps** (`MAX_IMAGE_BYTES`/`MAX_CAPTION_BYTES`); **one entry per peer** (block-0-only download + dedup bounds the gallery); and keeps the tip `address` only if the entry's `burn` attestation names that wallet (`tronAddress === address`) so **every tip goes to the wallet that paid in**. Same set of cores in → byte-identical gallery out on every peer (the CRDT property — no indexer, no consensus). (3) There is **no shared gallery key**: each participant owns its core, whose key is self-certified by its join attestation, so a relay can't redirect peers to a rogue gallery (and there's nothing to sign). With no wallet (tests/headless) ingest is join-attestation-only and addresses are empty. Gotcha learned: capture the burn proof + join sig into locals **before** the `postEntry` append `await` (`feed-crdt.js`) — a fast wave can end mid-append and a new wave's `clearBurnProof` would blank them, stripping your own tip address (the proof deliberately survives the end-of-wave reset; see `selfie.js`).

**Gotchas learned:** (1) `package.json#upgrade` must be a valid `pear://` link (mint via `pear touch`) or `electron-forge start` refuses to boot. (2) Cold discovery on a fresh public-DHT topic takes **~20–35s** — launch demo instances early; consider a well-known bootstrap peer's `joinPeer` key or a local DHT bootstrap to speed it up. (3) Two Hyperswarm instances in one process don't reliably discover each other; always one instance per process — so a host that ALSO uses Hyperswarm should pass its instance to `createEngine({ swarm })` / `createWave({ swarm })` (the engine shares it and never destroys it; on close it only leaves the topics it joined + detaches its listeners) rather than let the engine open a second. (4) A local DHT (`@hyperswarm/testnet` / `bootstrap` option) gives instant same-machine discovery and is confirmed working on the dev Mac (it only failed inside the build sandbox due to blocked loopback UDP) — use it for the live demo to avoid the ~30s public-DHT wait. (5) All instances on the same `topicId` share one ring; the app default (`hyperwave:demo:v1`) is fixed, so instances on the public DHT collide — isolate tests with `HYPERWAVE_TOPIC`/the `topicId` engine option or the local bootstrap. ## What This Project Is

A submission for the **Tether Developers Cup** (DoraHacks), theme "football / global tournament moment." It turns a stadium "Mexican wave" into a permissionless P2P relay:

- Peers join a match-specific Hyperswarm topic. Each peer's Noise public key deterministically maps to a fixed **position on the DHT ring** (angle derived from the key's top bytes) — the ring _is_ the stadium seating chart. No registration, no server.
- A **deterministic sweep** rolls the wave around that ring: the initiator floods a start time, and every joined peer fires at its own angle-ordered moment — like a real stadium wave, choreography rather than a passed object. The wave takes a fixed few seconds no matter how many peers; a dead peer's moment simply passes.
- Selfies are captured **during the lobby** and posted, when each peer's sweep slot fires, to a per-wave **multicore CRDT** gallery — one Hypercore per participant, merged locally into one ordered view (no indexer, no admission; writes gated on a signed join attestation + burn attestation).
- **WDK** powers all money: self-custodial wallets, burned participation fees, and gallery tipping.

## Core Design Rules (baked-in decisions — do not re-litigate)

- **The engine (`packages/hyperwave-engine`) is theme-agnostic; keep it that way.** It speaks generic concepts only — a **wave** that **sweeps** a **ring**, an **entry** with an **opaque `payload`** in a per-wave **feed**, a cosmetic per-peer **tag**, a shared **topic** (`topicId`). The football "stadium wave" (selfies, gallery, countries, the ⚽, kick-off) lives entirely in the desktop app (repo root) / `mobile/`, which map their concepts onto the engine's at the IPC boundary (`renderer/lib/ipc.js` + `app.js`: a selfie `{image,caption}` → the entry payload; a country → the tag; `set-tag`/`stage-entry`/`feed` on the wire). Do NOT reintroduce football vocabulary (selfie/gallery/country/kickoff/match) into the engine package.
- **Money model = burned fees + tips; NO sponsor rewards.** Payouts/rewards of any kind are sybil-farmable — do not add them. Fees are burned to Tron's black hole (nobody profits); tips are the only earnings.
- **No peer roles.** Every peer runs the same code; the only asymmetry is per-wave (the initiator archives its own wave's gallery).
- **The wave is a deterministic sweep, not a passed object.** `wave-start` carries `t0`+`lapMs`; every roster peer self-triggers at its angle-ordered slot (`sweep.js`) and self-completes at `t0+lapMs+grace`. No token, no healing, no completion message — wall-clock is a chosen constant regardless of N, and a dead peer's slot simply passes.
- **One wire encoding (JSON).** Every gossip message is `JSON.stringify`'d over a single `compact-encoding` **`string`** Protomux frame, and carries the uniform Ed25519 envelope (`origin`/`ts`/`sig`, protocol.md §5.0) verified at the receive edge — over Noise-encrypted Hyperswarm streams. Domain attestations (join/burn) sign their own tuples underneath.
- **Selfies never block the wave.** Capture happens in the lobby (before the sweep); the staged frame posts when the peer's slot fires. Any experiential pacing is a renderer concern, not the protocol.
- **Multicore CRDT gallery — no admission, no indexer.** Each participant owns one Hypercore and posts its single selfie at block 0; its key rides its own `wave-join` (self-certified by the join attestation). Every peer ingests every join, opens that core, downloads block 0 only, and merges locally (`mergeFeed`) — same cores in, byte-identical gallery out. The gate is signature-only (`verifyJoin`, plus `burnAuthorizes` when paid) with no on-chain call on the ingest path; the burn is verified where it pays off (tippers/auditors via `burnTx`). Spam is bounded by block-0-only download → one entry per peer + byte caps. A join that misses the lobby spectates.
- **Testnet only.** The DESKTOP default is **Cashu** (testnet ecash on a Lightning mint; the dev default is the free auto-paying `testnut` mint). The Tron wallet uses native TRX on Nile (not TRC-20 USDT — a TRX transfer pays its own fee from the same balance). No smart contracts.

## Platform

Two hosts over one engine (`packages/hyperwave-engine`):

- **Desktop (primary): Electron forked from [`holepunchto/hello-pear-electron`](https://github.com/holepunchto/hello-pear-electron)** — run N peers on one laptop via `--storage <dir>`, reliable webcam via `getUserMedia`. Three-process split (respect it): **renderer** (Chromium, UI only, never touches the swarm), **Electron main** (window/lifecycle/IPC relay), **Bare worker** (all Holepunch P2P + WDK; a thin host over the engine's `createEngine()`).
- **Mobile: Expo + react-native-bare-kit** (`mobile/`) — same engine as a Bare worklet, same JSON IPC surface. Runs on the iOS simulator; rich UI/camera still open.
- **Spectator is just a peer instance.** The webcam selfie is the proof-of-humanity (no sensor layers).

## Stack

- **App shell:** Electron via hello-pear-electron (Electron Forge; `pear-runtime` for OTA updates).
- **Networking:** Hyperswarm (DHT discovery, NAT hole-punching, Noise duplex streams) — in the Bare worker.
- **Shared state:** a multicore CRDT selfie gallery — one Hypercore per participant (merged locally, no Autobase/indexer; images inline as dataURLs — no Hyperblobs) over Corestore, in the worker.
- **Payments:** the engine talks to payments through the abstract **`Wallet` base class** (`wallet.js`) — the concrete interface (`type`/`fee`/`address`/`balances`/`send`/`burn`/`verifyBurnTx`/`transactions`/`dispose`); an app plugs in its own by injecting a factory returning a `Wallet` subclass (`createEngine({ deps: { createPayments } })`). The default **`TronWallet`** (`tron-wallet.js`) = WDK, self-custodial, **native TRX on Tron Nile**, plain transfers (no contracts); runs **in the Bare worker** (WDK supports Bare). Packages: `@tetherto/wdk`, `@tetherto/wdk-wallet-tron`. WDK is ESM-only → `tron-wallet.js` bridges via dynamic `import()`. The wallet-agnostic fee flows live in `payments.js`. Each wallet declares a `type` (`'tron-nile'`) that rides paid waves' announces so a joiner only joins a wave whose payment mechanism it supports.

## Code Style (beyond prettier/lunte — apply to all new and edited code)

- **Always brace `if`/`else`/`for`/`while` bodies** — no brace-less single-statement bodies, even one-liners.
- **No single-letter / cryptic variable names** (`s`, `f`, `r`, `t`, `pt`, `it`, `k`…). Use descriptive names. Acceptable idioms: loop index `i`, geometry locals (`dx`/`dy`/`x`/`y`), `ctx` (canvas), `ev`/`evt` (events).
- **Avoid `switch` statements where possible** — prefer a lookup table (object keyed by case → value or handler function).
- **Avoid deeply nested `if`/`else`** — prefer early returns / guard clauses; extract a helper when branching gets deep. Same spirit applies to nested ternaries.
- **One statement per line** — never `a; b` on one line or comma-operator statement chaining.
- **Always use semicolons** — enforced by prettier (`.prettierrc.js` overrides holepunch's `semi: false` with `semi: true`).
- **80-character line limit** — enforced by prettier (`.prettierrc.js` overrides holepunch's `printWidth: 100` with `printWidth: 80`).
- **At most two chained calls per line** — `funcA().funcB()` is fine; `funcA().funcB().funcC()` is not. For 3+ calls, each call goes on its own indented line under the receiver. Caveat: prettier only produces that broken form when the chain exceeds 80 chars — it collapses a manually broken short chain back onto one line (no config option controls this) — so for a 3+ chain that fits in 80 chars, split it with an intermediate variable instead (`const hex = BigInt(nid).toString(16); return hex.padStart(16, '0');`). Not prettier-enforceable; apply by convention.
- **Always strict equality** — `===`/`!==`, never `==`/`!=`.
- **Declare all variables at the top of their scope** — module state/DOM refs/config consts at the top of the module, function locals at the top of the function (guard clauses may precede them). Function and handler-table definitions are not "variables" and live in their sections.
- **Never `setInterval`** — use a self-rescheduling `setTimeout`: the callback does its work, then re-arms itself (`timer = setTimeout(tick, ms)`) as its **last** step. Ticks can never overlap/stack, a slow or async tick delays the next instead of racing it, and each tick can decide not to continue. Clear with `clearTimeout`; the handle changes every tick, so keep it in a `let` and re-check it after async work before re-arming.
- **Google TypeScript Style Guide** (https://google.github.io/styleguide/tsguide.html) is the baseline for anything not covered above, where it applies to plain JS and doesn't conflict with this list or the prettier config. Notably: named exports only (no `export default`; exception: framework-required, e.g. the Expo root component), `const`/`let` never `var`, `CONSTANT_CASE` for module-level constants, `camelCase`/`PascalCase` otherwise, prefer `for…of`.
- **No functions with more than 3 positional parameters** — take a single destructured options object instead (`function myFunc({ waveId, hopCount, chainHash, sigHex })`; defaults inline, as in `finishWave(waveId, { stalled = false } = {})`). The real hazard is adjacent same-typed args (hex strings especially) silently transposed at a call site, so prefer the object form even at 2–3 args when the types are interchangeable. **Enforced in `packages/hyperwave-engine/`; advisory elsewhere.** Exemption: geometry/canvas draw helpers (renderer), where positional `(x, y, r, …)` is the universal idiom.
- **No pass-through `return func()` unless the return value is meant** — only write `return func()` when the caller's contract is to return `func()`'s value/type. If the call is done just for its effect (or the function returns nothing meaningful), call it on its own line and `return;` separately, so the function doesn't accidentally leak a return value it never promised.
- **Class private members use `#`, not `_`** — real ES private fields/methods (`#waiters`, `#ingest()`), the modern way, enforced by the language rather than convention. (An `_` prefix remains fine for its _other_ job: marking deliberately-unused parameters, e.g. `(_evt, url) =>`. Third-party internals like WDK's `_tronWeb` are theirs to name.)

## Message Types

As built (`packages/hyperwave-engine/docs/protocol.md` §5 has the schemas) — every message carries a **uniform envelope** (`origin` = the author's ring id, `ts` = author timestamp, `sig` = Ed25519 by `origin` over the whole message minus sig; `mid` too on flooded kinds) stamped at wave.js's `originate()` choke point and verified at the receive edge (sig + `ts` age-drop → replay/flood-circulation cap) — `origin` is the author on every kind (no per-kind `id`/`by`/`peerId`), the only exception being wave-sync's `by` (the wave initiator, distinct from the sync's sender). Eight kinds: `heartbeat` (liveness + tag, one hop to every connection), `subs` (this peer's subscription set — the waveIds it holds cores for; one hop, on connect + on change; a neighbour scopes which waves' join/start/sync it forwards here — scaling.md Phase 3), `wave-announce` (flooded across the directory; `waveId` + `by` + `lobbyMs` + the kick-off `paid` proof + optional `walletType` (the payment-mechanism id, so joiners decide if they can pay) + optional `fee` (the **initiator-set** participation fee every joiner burns; a peer refuses a wave whose fee is below its local `minFee` floor) — **no feed key**, none exists; a unicast `catchup` re-announce on connect lets a late peer discover older waves), `wave-join` (flooded **only to a wave's subscribers**; publishes the peer's own feed core: `peerId` + `writerKey` + `joinSig` + optional `burn`), `wave-start` (flooded to subscribers; the full `writers` set `[{peerId, writerKey, joinSig}]` + `t0` + `lapMs` + `paid` + `walletType` + `fee` — the roster is derived as `{by} ∪ writers[].peerId`, no separate roster field, no key; the roster is **capped at `MAX_WRITERS`=256** so this O(N) payload is bounded — a peer arriving once a wave is full spectates, `join-blocked reason:'roster-full'`; the receive edge also drops any gossip frame > `MAX_FRAME_BYTES`=256KB before parsing), `wave-sync` (catch-up, unicast to a mutually-subscribed neighbour; `phase` + `writers` + `t0` + `lapMs` + `paid` + `walletType` + `fee` + `lobbyMsLeft`), `wave-note` (a roster member's **authenticated broadcast** — an opaque size-capped `note` flooded to a wave's subscribers, but **relayed/processed only if `origin` is in that wave's roster** (`wave.writers`); the engine's generic "announcement from a participant" primitive, theme-agnostic — surfaced as a `note` engine event), and `wave-dm` (the **directed/private counterpart of wave-note** — the same opaque `note` sent UNICAST to one peer `to` over a direct connection; NOT flooded (no `mid`), so the receive-edge identity rule forces `origin`==sender = no relay; delivered over an existing direct channel or via `swarm.joinPeer(peerId)` dial-and-flush; surfaced as a `dm` event. Its use: **private Cashu tip delivery** — the bearer token is unicast to the recipient instead of flooded, so the token + who-tipped-whom stay off the mesh (Cashu unlinkability at the network layer); the desktop then floods only a **stripped** `wave-note` `{kind:'tip', amount}` for social proof. A chain (Tron) tip stays a full public `wave-note` `{kind:'tip', to, peerId, amount, hash}`). All ride one Protomux channel per connection. Feed entries are `wave-entry` ops at block 0 of each participant's own core (not on-wire messages); the ⚽ position is renderer-local (schedule-driven), not gossip.
