The MVP is feature-complete. `apps/desktop/` is the Electron app (forked from hello-pear-electron). Built: discovery, **heartbeat gossip** (Protomux channel; a single `heartbeat` message carries liveness + country ‚Äî no ring structure), live sorted ring, the **deterministic sweep** (`wave-start` carries `t0` + `lapMs`; every roster peer derives the identical angle-ordered schedule from the canonical roster ‚Äî pure `sweep.js` ‚Äî and self-triggers at its own slot; the wave ends deterministically at `t0+lapMs+grace` on every peer, no completion message, no healing; a dead peer's slot just passes), and the **multicore CRDT selfie gallery** (`feed-crdt.js`) ‚Äî each participant owns ONE Hypercore in the per-wave namespace `wave-feed:<waveId>` and appends its single `wave-entry` (inline JPEG thumbnail) at its sweep slot; its core key rides its `wave-join` (self-certified by the **join attestation** ‚Äî `attest.js signJoin` over `waveId|peerId|writerKey`), so **every peer** ingests every `wave-join`, opens that core (`addWriter`), and merges the set locally (`mergeFeed`) ‚Äî no indexer, no admission, no shared gallery key. wave-start/wave-sync carry the full `writers` set so a late adopter is self-contained. Each subscribed wave also joins its own sub-topic `hash(prefix:topic:wave)` for participant discovery off the O(N) mesh. (Directory-at-scale indexing is Phase 4.) **Only opted-in roster members selfie**; everyone renders the sweep. ## What This Project Is

A submission for the **Tether Developers Cup** (DoraHacks), theme "football / global tournament moment." It turns a stadium "Mexican wave" into a permissionless P2P relay:

- Peers join a match-specific Hyperswarm topic. Each peer's Noise public key deterministically maps to a fixed **position on the DHT ring** (angle derived from the key's top bytes) ‚Äî the ring _is_ the stadium seating chart. No registration, no server.
- A **deterministic sweep** rolls the wave around that ring: the initiator floods a start time, and every joined peer fires at its own angle-ordered moment ‚Äî like a real stadium wave, choreography rather than a passed object. The wave takes a fixed few seconds no matter how many peers; a dead peer's moment simply passes.
- Selfies are captured **during the lobby** and posted, when each peer's sweep slot fires, to a per-wave **multicore CRDT** gallery ‚Äî one Hypercore per participant, merged locally into one ordered view (no indexer, no admission; writes gated on a signed join attestation + burn attestation).
- **WDK** powers all money: self-custodial wallets, burned participation fees, and gallery tipping.

## Core Design Rules (baked-in decisions ‚Äî do not re-litigate)

- **The engine (`packages/hyperwave-engine`) is theme-agnostic; keep it that way.** It speaks generic concepts only ‚Äî a **wave** that **sweeps** a **ring**, an **entry** with an **opaque `payload`** in a per-wave **feed**, a cosmetic per-peer **tag**, a shared **topic** (`topicId`). The football "stadium wave" (selfies, gallery, countries, the ‚öΩ, kick-off) lives entirely in `apps/desktop` / `apps/mobile`, which map their concepts onto the engine's at the IPC boundary (`renderer/lib/ipc.js` + `app.js`: a selfie `{image,caption}` ‚Üí the entry payload; a country ‚Üí the tag; `set-tag`/`stage-entry`/`feed` on the wire). Do NOT reintroduce football vocabulary (selfie/gallery/country/kickoff/match) into the engine package.
- **Money model = burned fees + tips; NO sponsor rewards.** Payouts/rewards of any kind are sybil-farmable ‚Äî do not add them. Fees are burned to Tron's black hole (nobody profits); tips are the only earnings.
- **No peer roles.** Every peer runs the same code; the only asymmetry is per-wave (the initiator archives its own wave's gallery).
- **The wave is a deterministic sweep, not a passed object.** `wave-start` carries `t0`+`lapMs`; every roster peer self-triggers at its angle-ordered slot (`sweep.js`) and self-completes at `t0+lapMs+grace`. No token, no healing, no completion message ‚Äî wall-clock is a chosen constant regardless of N, and a dead peer's slot simply passes.
- **One wire encoding (JSON).** Every gossip message is `JSON.stringify`'d over a single `compact-encoding` **`string`** Protomux frame, and carries the uniform Ed25519 envelope (`origin`/`ts`/`sig`, protocol.md ¬ß5.0) verified at the receive edge ‚Äî over Noise-encrypted Hyperswarm streams. Domain attestations (join/burn) sign their own tuples underneath.
- **Selfies never block the wave.** Capture happens in the lobby (before the sweep); the staged frame posts when the peer's slot fires. Any experiential pacing is a renderer concern, not the protocol.
- **Multicore CRDT gallery ‚Äî no admission, no indexer.** Each participant owns one Hypercore and posts its single selfie at block 0; its key rides its own `wave-join` (self-certified by the join attestation). Every peer ingests every join, opens that core, downloads block 0 only, and merges locally (`mergeFeed`) ‚Äî same cores in, byte-identical gallery out. The gate is signature-only (`verifyJoin`, plus `burnAuthorizes` when paid) with no on-chain call on the ingest path; the burn is verified where it pays off (tippers/auditors via `burnTx`). Spam is bounded by block-0-only download ‚Üí one entry per peer + byte caps. A join that misses the lobby spectates.
- **Testnet only.** Native TRX on Nile (not TRC-20 USDT ‚Äî a TRX transfer pays its own fee from the same balance). No smart contracts.

## Platform

Two hosts over one engine (`packages/hyperwave-engine`):

- **Desktop (primary): Electron forked from [`holepunchto/hello-pear-electron`](https://github.com/holepunchto/hello-pear-electron)** ‚Äî run N peers on one laptop via `--storage <dir>`, reliable webcam via `getUserMedia`. Three-process split (respect it): **renderer** (Chromium, UI only, never touches the swarm), **Electron main** (window/lifecycle/IPC relay), **Bare worker** (all Holepunch P2P + WDK; a thin host over the engine's `createEngine()`).
- **Mobile: Expo + react-native-bare-kit** (`apps/mobile/`) ‚Äî same engine as a Bare worklet, same JSON IPC surface. Runs on the iOS simulator; rich UI/camera still open.
- **Spectator is just a peer instance.** The webcam selfie is the proof-of-humanity (no sensor layers).

## Stack

- **App shell:** Electron via hello-pear-electron (Electron Forge; `pear-runtime` for OTA updates).
- **Networking:** Hyperswarm (DHT discovery, NAT hole-punching, Noise duplex streams) ‚Äî in the Bare worker.
- **Shared state:** a multicore CRDT selfie gallery ‚Äî one Hypercore per participant (merged locally, no Autobase/indexer; images inline as dataURLs ‚Äî no Hyperblobs) over Corestore, in the worker.
- **Payments:** the engine talks to payments through the abstract **`Wallet` base class** (`wallet.js`) ‚Äî the concrete interface (`type`/`fee`/`address`/`balances`/`send`/`burn`/`verifyBurnTx`/`transactions`/`dispose`); an app plugs in its own by injecting a factory returning a `Wallet` subclass (`createEngine({ deps: { createPayments } })`). The default **`TronWallet`** (`tron-wallet.js`) = WDK, self-custodial, **native TRX on Tron Nile**, plain transfers (no contracts); runs **in the Bare worker** (WDK supports Bare). Packages: `@tetherto/wdk`, `@tetherto/wdk-wallet-tron`. WDK is ESM-only ‚Üí `tron-wallet.js` bridges via dynamic `import()`. The wallet-agnostic fee flows live in `payments.js`. Each wallet declares a `type` (`'tron-nile'`) that rides paid waves' announces so a joiner only joins a wave whose payment mechanism it supports.

## Code Style (beyond prettier/lunte ‚Äî apply to all new and edited code)

- **Always brace `if`/`else`/`for`/`while` bodies** ‚Äî no brace-less single-statement bodies, even one-liners.
- **No single-letter / cryptic variable names** (`s`, `f`, `r`, `t`, `pt`, `it`, `k`‚Ä¶). Use descriptive names. Acceptable idioms: loop index `i`, geometry locals (`dx`/`dy`/`x`/`y`), `ctx` (canvas), `ev`/`evt` (events).
- **Avoid `switch` statements where possible** ‚Äî prefer a lookup table (object keyed by case ‚Üí value or handler function).
- **Avoid deeply nested `if`/`else`** ‚Äî prefer early returns / guard clauses; extract a helper when branching gets deep. Same spirit applies to nested ternaries.
- **One statement per line** ‚Äî never `a; b` on one line or comma-operator statement chaining.
- **Always use semicolons** ‚Äî enforced by prettier (`.prettierrc.js` overrides holepunch's `semi: false` with `semi: true`).
- **80-character line limit** ‚Äî enforced by prettier (`.prettierrc.js` overrides holepunch's `printWidth: 100` with `printWidth: 80`).
- **At most two chained calls per line** ‚Äî `funcA().funcB()` is fine; `funcA().funcB().funcC()` is not. For 3+ calls, each call goes on its own indented line under the receiver. Caveat: prettier only produces that broken form when the chain exceeds 80 chars ‚Äî it collapses a manually broken short chain back onto one line (no config option controls this) ‚Äî so for a 3+ chain that fits in 80 chars, split it with an intermediate variable instead (`const hex = BigInt(nid).toString(16); return hex.padStart(16, '0');`). Not prettier-enforceable; apply by convention.
- **Always strict equality** ‚Äî `===`/`!==`, never `==`/`!=`.
- **Declare all variables at the top of their scope** ‚Äî module state/DOM refs/config consts at the top of the module, function locals at the top of the function (guard clauses may precede them). Function and handler-table definitions are not "variables" and live in their sections.
- **Never `setInterval`** ‚Äî use a self-rescheduling `setTimeout`: the callback does its work, then re-arms itself (`timer = setTimeout(tick, ms)`) as its **last** step. Ticks can never overlap/stack, a slow or async tick delays the next instead of racing it, and each tick can decide not to continue. Clear with `clearTimeout`; the handle changes every tick, so keep it in a `let` and re-check it after async work before re-arming.
- **Google TypeScript Style Guide** (https://google.github.io/styleguide/tsguide.html) is the baseline for anything not covered above, where it applies to plain JS and doesn't conflict with this list or the prettier config. Notably: named exports only (no `export default`; exception: framework-required, e.g. the Expo root component), `const`/`let` never `var`, `CONSTANT_CASE` for module-level constants, `camelCase`/`PascalCase` otherwise, prefer `for‚Ä¶of`.
- **No functions with more than 3 positional parameters** ‚Äî take a single destructured options object instead (`function myFunc({ waveId, hopCount, chainHash, sigHex })`; defaults inline, as in `finishWave(waveId, { stalled = false } = {})`). The real hazard is adjacent same-typed args (hex strings especially) silently transposed at a call site, so prefer the object form even at 2‚Äì3 args when the types are interchangeable. **Enforced in `packages/hyperwave-engine/`; advisory elsewhere.** Exemption: geometry/canvas draw helpers (renderer), where positional `(x, y, r, ‚Ä¶)` is the universal idiom.
- **No pass-through `return func()` unless the return value is meant** ‚Äî only write `return func()` when the caller's contract is to return `func()`'s value/type. If the call is done just for its effect (or the function returns nothing meaningful), call it on its own line and `return;` separately, so the function doesn't accidentally leak a return value it never promised.
- **Class private members use `#`, not `_`** ‚Äî real ES private fields/methods (`#waiters`, `#ingest()`), the modern way, enforced by the language rather than convention. (An `_` prefix remains fine for its _other_ job: marking deliberately-unused parameters, e.g. `(_evt, url) =>`. Third-party internals like WDK's `_tronWeb` are theirs to name.)

## Message Types

As built (`packages/hyperwave-engine/docs/protocol.md` ¬ß5 has the schemas) ‚Äî every message carries a **uniform envelope** (`origin` = the author's ring id, `ts` = author timestamp, `sig` = Ed25519 by `origin` over the whole message minus sig; `mid` too on flooded kinds) stamped at wave.js's `originate()` choke point and verified at the receive edge (sig + `ts` age-drop ‚Üí replay/flood-circulation cap) ‚Äî `origin` is the author on every kind (no per-kind `id`/`by`/`peerId`), the only exception being wave-sync's `by` (the wave initiator, distinct from the sync's sender). Six kinds: `heartbeat` (liveness + tag, one hop to every connection), `subs` (this peer's subscription set ‚Äî the waveIds it holds cores for; one hop, on connect + on change; a neighbour scopes which waves' join/start/sync it forwards here ‚Äî scaling.md Phase 3), `wave-announce` (flooded across the directory; `waveId` + `by` + `lobbyMs` + the kick-off `paid` proof + optional `walletType` (the payment-mechanism id, so joiners decide if they can pay) + optional `fee` (the **initiator-set** participation fee every joiner burns; a peer refuses a wave whose fee is below its local `minFee` floor) ‚Äî **no feed key**, none exists; a unicast `catchup` re-announce on connect lets a late peer discover older waves), `wave-join` (flooded **only to a wave's subscribers**; publishes the peer's own feed core: `peerId` + `writerKey` + `joinSig` + optional `burn`), `wave-start` (flooded to subscribers; the full `writers` set `[{peerId, writerKey, joinSig}]` + `t0` + `lapMs` + `paid` + `walletType` + `fee` ‚Äî the roster is derived as `{by} ‚à™ writers[].peerId`, no separate roster field, no key; the roster is **capped at `MAX_WRITERS`=256** so this O(N) payload is bounded ‚Äî a peer arriving once a wave is full spectates, `join-blocked reason:'roster-full'`; the receive edge also drops any gossip frame > `MAX_FRAME_BYTES`=256KB before parsing), `wave-sync` (catch-up, unicast to a mutually-subscribed neighbour; `phase` + `writers` + `t0` + `lapMs` + `paid` + `walletType` + `fee` + `lobbyMsLeft`). All ride one Protomux channel per connection. Feed entries are `wave-entry` ops at block 0 of each participant's own core (not on-wire messages); the ‚öΩ position is renderer-local (schedule-driven), not gossip.
